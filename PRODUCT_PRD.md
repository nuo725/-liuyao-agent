# 宽窄·Orbit 产品需求文档（PRD）

版本：v1.0  
日期：2026-06-04  
状态：产品基线  
适用范围：Flutter 客户端、业务后端、独立六爻 Agent 后端

## 1. 文档目的

本文档以当前已完成的前端 UI、交互流程和模块 API 文档为事实基线，定义宽窄·Orbit 的产品定位、目标架构、后端职责、Agent 能力、待补全需求和验收标准。

本文档不重新设计现有前端信息架构。以下内容视为固定产品事实：

- 底部主导航为 `宽窄之间`、`此刻`、`活动`、`我的`。
- 底部中央无文字主按钮用于进入 `问一问` 仪式与 Agent 解读流程。
- `消息中心`不属于底部主导航，从 `宽窄之间`顶部通知入口进入。
- `宽窄之间`现有信息流仅包含 `推荐`与 `深谈`两个页签。
- 产品当前为 UI 模拟优先阶段，真实业务后端尚未完整接入。

## 2. 产品概述

### 2.1 产品名称

- 中文品牌：宽窄·Orbit
- 核心社区名称：宽窄之间
- 核心 Agent 入口：问一问

### 2.2 产品定位

宽窄·Orbit 是一个以六爻 Agent 为内容与回应引擎、以情绪陪伴为体验底色、以社群互动为长期关系场的社区型 APP。

用户可以带着真实困惑进入 `问一问`，通过起卦与 Agent 解读获得一次有结构、有温度的自我梳理；也可以在 `宽窄之间`阅读、发布和讨论他人的感受与经历，在 `此刻`发现同频的人与历史故事，通过 `活动`建立更具体的参与关系，并在 `我的`中沉淀个人记录、互动和成长轨迹。

产品不是单纯的算卦工具，也不是普通聊天机器人。六爻是 Agent 的内部解释框架，社群是产品产生连接、留存和内容价值的核心场域。

### 2.3 产品表达原则

- 以心为引，映照而非解答。
- 保留六爻内核，不使用宿命化、恐吓式或绝对化表达。
- 不输出吉凶评分、运势评级或结果承诺。
- 不替用户做重大人生决定。
- 鼓励用户回到现实信息、可执行行动和真实关系中。
- 社区内容以共鸣、理解、经验分享和深度讨论为主。

## 3. 产品愿景与北极星目标

### 3.1 产品愿景

帮助用户在面对不确定、焦虑、关系困扰和人生选择时，通过六爻解读、温和陪伴和社群共鸣，完成一次安全、可表达、可被回应、可沉淀的自我梳理。

### 3.2 北极星指标

**周有效共鸣用户数（Weekly Meaningful Resonance Users，WMRU）**

满足以下任一条件的周去重用户计为有效共鸣用户：

1. 完成一次 `问一问` 解读，并产生保存、追问、生成解读卡、发布到社区等至少一个后续行为。
2. 在 `宽窄之间`产生发布、有效评论、收藏、关注等至少一个深度互动行为。
3. 在 `此刻`完成同频解锁并进入同频内容，或在 `活动`中完成报名参与。

该指标同时覆盖 Agent 价值、社区连接和长期参与，不以单纯起卦次数或信息流浏览量作为产品成功标准。

## 4. 目标用户

### 4.1 核心用户

- 18 至 35 岁，对传统文化、玄学、心理陪伴、情绪表达或深度社交感兴趣的用户。
- 正在经历感情、事业、情绪、选择、人际关系等困惑，希望获得新的观察角度。
- 愿意表达真实感受，但不一定愿意在熟人社交平台公开。
- 喜欢阅读他人的真实经历、深度讨论和后续变化。

### 4.2 典型需求

- “我现在很乱，希望有人帮我把问题梳理清楚。”
- “我不一定需要确定答案，但想知道还能从什么角度看这件事。”
- “我想看看是否有人也经历过类似的状态。”
- “我想记录自己一段时间内反复出现的问题和情绪。”
- “我想找到一些能认真交流，而不是只给结论的人。”

### 5.1 启动与首次体验

状态：`已完成 UI`

- 启动页展示宽窄·Orbit 品牌与星河视觉。
- 首次启动进入两页引导流程。
- 用户可以开始第一次完整体验，也可以跳过并登录。
- 第一次体验会自动进入 `问一问`，从问题输入开始，不自动跳过提问环节。
- 登录后可同步记录、互动与权益状态。

### 5.3 宽窄之间

状态：`已完成主要 UI，待真实后端接入`

已完成能力：

- `推荐`与 `深谈`双页签信息流。
- 图文内容、纯文本内容和解读卡关联内容展示。
- 作者信息、头像、用户名、显示 Handle、关注入口。
- 点赞、收藏、浏览量、评论入口。
- 帖子详情页。
- 评论弹层与本地评论模拟。
- 发布页，支持文本、图片和解读卡分享。
- 搜索入口、发布入口、通知入口。
- 举报、不感兴趣、屏蔽用户等操作入口。
- 加载、空状态、错误状态和加载更多状态。

待后端接入：

- 推荐流与深谈流分页。
- 帖子详情、评论线程、点赞、收藏、关注、举报、隐藏、屏蔽。
- 图片上传与 CDN 地址。
- 搜索帖子、用户和活动。
- 通知弹层与消息中心数据。
- 作者公开主页。
- 发布后跨设备同步与状态一致性。

产品要求：

- `推荐`面向较广泛的真实表达、生活感受和可共鸣内容。
- `深谈`面向更长文本、更完整的背景、更有讨论价值的解读与反思。
- 信息流不展示吉凶、评分、运势排名等字段。
- 内容应允许纯文本发布，不强制所有帖子绑定解读卡。
- 绑定解读卡的帖子只能使用脱敏后的公开版本，不得暴露用户原始私密问题。

### 5.4 此刻

状态：`已完成主要 UI，待真实后端接入`

已完成能力：

- 星空广场与轨道视觉。
- 摇一摇解锁交互。
- 同频入口解锁状态。
- 同频用户与历史同频两个内容页签。
- 基于六段结构签名的本地匹配模拟。

待后端接入：

- 设备级或账号级解锁状态。
- 同频用户和历史同频分页数据。
- 今日签名与匹配理由。
- 访客与登录用户的差异化内容范围。

产品要求：

- `此刻`不是第二个普通信息流，而是提供轻量、仪式化、带发现感的同频入口。
- 同频匹配用于建立共鸣，不用于给用户贴固定人格标签。
- 不展示匹配分数、优劣排序或命运绑定结论。

### 5.8 问一问与仪式流程

状态：`已完成核心 UI 与本地模拟，Agent 接入路径存在，待生产化`

当前完整流程：

1. 用户输入问题并选择问题分类。
2. 用户通过软件模式或硬件模式完成六段结构。
3. 系统进入悬置等待状态。
4. Agent 或本地降级逻辑生成解读。
5. 用户查看解读、继续追问或生成解读卡。
6. 用户可以保存、分享外部或发布到宽窄之间。
7. 会话进入收束阶段并保存到心绪存档。

当前问题分类：

- 关系
- 事业
- 情绪
- 选择
- 其他

当前前端已包含：

- 首次体验预览与登录门禁。
- 软件起卦和硬件入口。
- 解读加载、错误、超时和重试恢复模拟。
- 流式打字机效果。
- 继续追问。
- 每日解读与追问额度。
- 解读卡生成、主题选择、背景选择、保存、外部分享和发布。

## 6. 目标系统架构

### 6.1 架构组成

目标系统由三个独立层次组成：

1. Flutter 客户端  
   负责展示、交互、动画、输入采集、本地缓存和弱网体验。

2. 业务后端  
   负责用户、社区、活动、记录、通知、权益、审核、会话编排和数据一致性。

3. 独立六爻 Agent 后端  
   负责六爻结构分析、问题理解、情绪陪伴、解读生成、追问生成和输出安全处理。

### 6.2 生产数据流

生产环境中，客户端应优先调用业务后端，不直接依赖 Agent 后端：

1. 客户端提交问题、分类、六段结构和动爻信息。
2. 业务后端校验用户身份、额度、输入安全和幂等键。
3. 业务后端创建权威仪式会话与卦象记录。
4. 业务后端使用服务身份调用独立 Agent 后端。
5. Agent 后端返回结构化解读或流式内容。
6. 业务后端执行输出安全检查、持久化和额度结算。
7. 业务后端将结果流式返回客户端。
8. 用户发布到社区时，业务后端生成或校验脱敏公开版本。

### 6.3 服务职责边界

业务后端负责：

- 用户认证、会话与协议存证。
- 社区帖子、评论、互动、关注、举报、屏蔽和推荐。
- 活动、报名和状态通知。
- 心绪存档、浏览记录、互动记录和公开主页。
- 解读会话持久化、额度、订单和会员权益。
- 媒体上传、通知、审核、审计与数据删除。
- Agent 调用编排、超时、重试、降级和结果缓存。

独立 Agent 后端负责：

- 根据问题、分类、六段结构、动爻和时间上下文进行六爻分析。
- 生成首轮解读、澄清问题、追问回复和解读卡公开摘要。
- 识别高风险问题并执行安全降级。
- 保持同一会话内的上下文一致性。
- 返回稳定、可解析、可审计的结构化数据。

独立 Agent 后端不负责：

- 用户登录与社区账号。
- 点赞、评论、关注和信息流推荐。
- 订单、会员、额度账本。
- 直接保存用户完整隐私资料。
- 直接面向社区公开发布内容。

## 7. 六爻 Agent 产品需求

### 7.1 核心原则

- Agent 的内部推理核心仍然是六爻。
- 用户可见表达必须进行敏感处理，避免迷信化和宿命化。
- Agent 的价值是帮助用户整理局面、看见关系与行动空间，不是承诺预测准确率。
- 对用户的情绪进行安抚，但不制造依赖，不暗示只有继续起卦才能解决问题。

### 7.2 Agent 输入

必需字段：

- `sessionId`
- `question`
- `questionTag`
- `lines`
- `movingLines`
- `datetime`

可选字段：

- 用户当前情绪校准结果。
- 上一轮会话摘要。
- 用户追问内容。
- 客户端语言与版本。
- 是否需要公开分享版本。

不得传入：

- 与当前解读无关的完整用户隐私资料。
- 社区私信、通讯录或其他不必要的敏感数据。

### 7.3 Agent 输出

输出必须能够映射到当前前端 `InterpretationCard` 和会话结构：

- `mode`
- `needsClarification`
- `session`
- `hexagramSummary`
- `summary`
- `body`
- `focusPoints`
- `afterglow`
- `followupDirections`
- `microActions`
- `visualBlocks`
- `riskLevel`
- `warnings`
- `communitySafeVersion`

其中：

- `summary`用于快速理解当前局面。
- `body`用于完整解读。
- `focusPoints`用于提炼值得关注的关系与矛盾。
- `afterglow`用于温和收束，不制造紧迫感。
- `followupDirections`用于引导用户继续梳理，而不是诱导重复占卜。
- `microActions`用于提供现实中可执行的小行动。
- `communitySafeVersion`用于发布到宽窄之间时的脱敏公开内容。

### 7.4 输出表达要求

允许表达：

- “从这组结构看，当前更像是……”
- “这个趋势值得留意，但仍需要结合现实信息判断。”
- “你可以先从一个可验证的小行动开始。”
- “这不是结论，而是帮助你整理局面的一个角度。”

禁止表达：

- “一定会”“必然会”“命中注定”。
- “必分手”“必破财”“必失败”等恐吓式结论。
- 将疾病、死亡、法律结果、投资收益作为确定预测。
- 鼓励用户依赖持续起卦代替现实沟通和专业帮助。
- 使用吉凶评分、幸运指数或命运等级。

### 7.5 澄清与高风险处理

以下情况应优先澄清，而不是直接生成完整解读：

- 问题过于模糊，无法判断用户关注点。
- 同时包含多个互不相关的问题。
- 用户要求绝对预测或结果保证。
- 用户问题依赖缺失的重要现实信息。

以下情况必须安全降级：

- 自伤、自杀、伤害他人。
- 医疗诊断与治疗决策。
- 法律责任与诉讼结论。
- 投资、赌博和高风险财务决策。
- 犯罪、报复、跟踪或控制他人的意图。

高风险场景中，Agent 应停止六爻结论式解读，提供现实安全建议，并在必要时提示寻求专业帮助。

### 7.6 首轮解读与追问

- 首轮解读支持 SSE 流式返回。
- 追问必须复用原会话，不重新生成无关卦象。
- 会话上下文应包含原始问题、卦象摘要、已生成解读和历史追问。
- 追问次数和每日额度由业务后端权威控制。
- 当前前端模型中的单会话最大追问次数为 5，生产策略可配置但不得由客户端单独决定。

## 8. 业务后端产品需求

### 8.1 用户与认证

优先级：P0

- 手机验证码登录。
- 微信、QQ 等社交登录。
- 游客体验与游客升级。
- Access Token 与刷新机制。
- 用户协议、隐私政策版本和同意时间存证。
- 登录后合并游客时期的心绪存档、互动和权益。
- 账号注销与冷静期。

### 8.2 仪式会话与记录

优先级：P0

- 创建、恢复和结束仪式会话。
- 保存问题、分类、六段结构、动爻、解读卡和追问记录。
- 支持首次体验预览与登录后完整解读。
- 支持解读与追问的 SSE 流式返回。
- 支持超时、重试、幂等和 Agent 不可用时的降级。
- 支持跨设备恢复心绪存档。

### 8.3 社区

优先级：P0

- 推荐流与深谈流。
- 帖子发布、详情、删除和状态管理。
- 纯文本、图片和解读卡关联帖子。
- 评论、点赞、收藏、关注、举报、隐藏和屏蔽。
- 作者公开主页。
- 搜索帖子、用户和活动。
- 内容分页和稳定排序。
- 发布、点赞、收藏、关注等写操作支持幂等。

### 8.4 社区内容安全

优先级：P0

- 发帖前文本安全检测。
- 解读卡公开版本脱敏。
- 隐私信息识别与提示。
- 评论区辱骂、骚扰、广告和引流检测。
- 举报处理与审核记录。
- 对“收费看卦”“大师引流”“改运消灾”等高风险商业内容进行限制。

### 8.5 此刻与同频

优先级：P1

- 保存设备或账号级同频解锁状态。
- 返回同频用户、历史同频和匹配理由。
- 支持游客有限浏览和登录用户完整浏览。
- 不返回匹配分数或人格优劣排序。

### 8.6 活动

优先级：P1

- 活动列表、详情、报名与报名状态。
- 报名幂等。
- 活动状态变更通知。
- 活动审核与参与人数统计。

### 8.7 通知与消息中心

优先级：P1

- 评论、点赞、收藏、关注、活动状态、系统通知和 Agent 任务完成通知。
- 未读数量。
- 单条已读、全部已读、删除和分页。
- 推送 Token 注册与注销。
- 弱网环境下的本地缓存与状态同步。

### 8.8 个人中心

优先级：P1

- 用户资料、头像、封面和设置。
- 心绪存档、互动记录和浏览记录。
- 签到日历与连续签到。
- 公开主页与分享链接。
- 帮助与反馈。

### 8.9 权益、额度与订单

优先级：P1

- 普通用户与 VIP 用户的每日解读、追问额度。
- 签到奖励。
- 额度消费幂等。
- 会员计划、订单创建、支付确认和订单查询。
- 支付结果必须以服务端确认为准。

商业化不得使用恐吓、结果承诺或迷信焦虑促进付费。

## 9. 当前接口现状与统一要求

项目当前存在三套接口口径：

1. Flutter 独立 Agent 调用：`/api/v1/liuyao/app/...`
2. Node mock 业务后端：`/v1/ceremony`、`/v1/feed`、`/v1/profile` 等
3. 模块交接文档目标契约：`/ritual`、`/community`、`/notifications` 等

生产接入前必须确定一套业务后端权威契约，并完成以下统一：

- 统一基础路径、响应结构、错误码和认证头。
- 统一 `ritual`与 `ceremony`命名。
- 统一 `community`与 `feed`命名。
- 统一帖子是否必须绑定解读卡。基于当前前端能力，目标契约应允许纯文本帖子，`cardId`应可为空。
- 统一解读卡、卦象摘要和 Agent 会话字段。
- 统一 SSE 事件类型和断线重连策略。
- 客户端生产环境不应直接持有 Agent 服务凭证。

## 11. 非功能需求

### 11.1 安全与隐私

- 用户原始问题默认私密。
- 发布到社区前必须生成或校验脱敏版本。
- 媒体上传必须校验文件类型、大小和内容安全。
- 敏感数据传输使用 HTTPS。
- Agent 调用使用服务间认证。
- 支持用户导出、删除和注销相关数据。

### 11.2 稳定性

- 发布、点赞、收藏、关注、报名、额度消费和订单创建支持幂等。
- Agent 超时不应导致重复扣除额度。
- SSE 断开后支持恢复或明确重试。
- 本地缓存不可覆盖服务端权威状态。
- 关键流程必须有降级与错误提示。

### 11.3 可观测性

- 全链路 Request ID。
- Agent 调用耗时、失败率、超时率和降级率监控。
- 社区审核命中率和举报处理时长监控。
- 核心漏斗与留存指标埋点。
- 不在日志中记录完整用户私密问题和敏感个人信息。

## 15. 验收标准

### 15.1 产品验收

- 用户从首次启动可完整完成一次 `问一问`体验。
- 用户可在解读后追问、生成解读卡并发布到 `宽窄之间`。
- 用户发布内容后可在推荐或深谈信息流中看到，并完成评论、点赞和收藏。
- 用户可从顶部通知入口进入消息中心，不依赖新增底部页签。
- 用户可在 `此刻`完成同频解锁并查看同频内容。
- 用户可在 `活动`查看详情并完成报名。
- 用户可在 `我的`查看心绪存档、互动、浏览记录和权益状态。

### 15.2 架构验收

- 业务后端是用户、社区、记录、额度和审核数据的权威来源。
- 独立 Agent 后端只承担六爻解读与情绪陪伴生成职责。
- 客户端生产环境通过业务后端调用 Agent 能力。
- 所有关键写操作具备幂等处理。
- Agent 不可用时不会重复扣除额度或破坏会话状态。

### 15.3 安全验收

- 用户原始问题默认不公开。
- 解读卡发布前完成脱敏。
- 高风险问题不会获得绝对预测或危险建议。
- 社区能够处理举报、屏蔽、广告引流和收费看卦内容。
- 用户可注销账号并触发数据处理流程。

## 16. 最终产品定义

宽窄·Orbit 的核心不是让用户获得一个确定答案，而是让用户在不确定中获得一次有结构的回应，并在社群中感到自己并不孤立。

`问一问`负责帮助用户看见当下。  
`宽窄之间`负责让真实表达被回应。  
`此刻`负责让用户发现同频连接。  
`活动`负责让连接走向参与。  
`我的`负责让经历被长期沉淀。

六爻 Agent 是内容与回应引擎，社群关系是产品长期价值的来源。
